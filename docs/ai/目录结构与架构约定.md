# 目录结构与架构约定

> 收录时间：2026-04  
> 覆盖：Bulletproof React、Feature-Sliced Design (FSD)、本项目结构对照、ESLint 导入约束

---

## 一、主流架构方案对比

| 方案 | 核心思想 | 分层方式 | 复杂度 | 适合规模 |
|------|---------|---------|--------|---------|
| **Bulletproof React** | Feature 模块化 + 单向依赖 | features / components / pages | 中 | 中大型 |
| **Feature-Sliced Design** | 7 层架构 + 领域切片 | app / pages / widgets / features / entities / shared | 高 | 大型 |
| **按文件类型** | components / hooks / utils / pages | 传统分类 | 低 | 小型 |
| **按路由** | pages/xxx 下放所有关联代码 | Next.js 风格 | 低 | 中型 |

## 二、Bulletproof React 结构（本项目基础）

> 来源：https://github.com/alan2207/bulletproof-react  
> 核心理念：**feature 自包含 + 单向依赖流**

```
src/
├── app/                  # 应用层：路由、Provider、入口
│   ├── routes/           # 路由配置
│   ├── app.tsx           # 主组件
│   ├── provider.tsx      # 全局 Provider 组合
│   └── router.tsx        # 路由声明
├── assets/               # 静态资源
├── components/           # 全局共享组件（无业务逻辑）
├── config/               # 全局配置、环境变量
├── features/             # ⭐ 业务功能模块（按领域拆分）
├── hooks/                # 全局共享 hooks
├── lib/                  # 第三方库预配置封装
├── stores/               # 全局状态 store
├── testing/              # 测试工具和 mock
├── types/                # 全局类型定义
└── utils/                # 全局工具函数
```

### Feature 内部结构

```
src/features/awesome-feature/
├── api/            # API 请求 + hooks
├── assets/         # 功能私有静态资源
├── components/     # 功能私有组件
├── hooks/          # 功能私有 hooks
├── stores/         # 功能私有状态
├── types/          # 功能私有类型
├── utils/          # 功能私有工具
└── index.ts        # 公共 API（只导出外部需要的）
```

### 单向依赖流

```
shared (components / hooks / utils / types)
         ↓ 可被所有层导入
features (auth / workbench / workspace)
         ↓ 可被 app 层导入
app (routes / providers / entry)

❌ features 之间不能互相导入
❌ shared 不能导入 features 或 app
❌ features 不能导入 app
```

## 三、Feature-Sliced Design (FSD)

> 来源：https://feature-sliced.design  
> 核心理念：**7 层标准化 + 领域切片 + 技术段落**

### 7 层结构

```
src/
├── app/            # 应用入口：路由、Provider、全局样式
├── pages/          # 页面级组件（组装 widgets/features）
├── widgets/        # 大型自包含 UI 块（整合 features + entities）
├── features/       # 用户可交互功能（带业务价值的行为）
├── entities/       # 业务实体（user / project / message）
├── shared/         # 通用复用（ui / api / lib / config）
└── (processes/)    # 已废弃：跨页面场景
```

### 层级规则

```
app → 可导入所有层
pages → 可导入 widgets / features / entities / shared
widgets → 可导入 features / entities / shared
features → 可导入 entities / shared
entities → 可导入 shared
shared → 不能导入任何上层

同层 Slice 之间不能互相导入！
```

### Slice 内部 Segments

```
features/auth/
├── ui/        # UI 组件
├── api/       # 后端交互
├── model/     # 数据模型、store、业务逻辑
├── lib/       # 本 slice 工具
└── config/    # 配置和 feature flags
```

## 四、本项目结构对照

> 当前采用 **Bulletproof React 简化版**

```
src/
├── components/                  # 可复用 UI 组件（与业务无关）
│   ├── ui/                      # 基础原子组件 (Button, Input, Badge…)
│   │   ├── Button/
│   │   │   ├── Button.tsx
│   │   │   ├── Button.test.tsx
│   │   │   └── index.ts
│   │   └── ...
│   ├── layout/                  # 布局组件 (AppShell, Sidebar, Header…)
│   ├── feedback/                # 反馈组件 (Toast, Modal, Tooltip…)
│   ├── data-display/            # 数据展示 (Table, Card, List…)
│   └── data-input/              # 数据录入 (Form, Select, DatePicker…)
├── features/                    # 业务功能模块（按领域划分）
│   ├── workbench/               # 工作台功能
│   │   ├── components/          # 该功能私有组件
│   │   ├── hooks/               # 该功能私有 hooks
│   │   ├── utils/               # 该功能私有工具
│   │   └── index.ts
│   └── workspace/               # 工作空间功能
├── hooks/                       # 全局可复用 hooks
├── utils/                       # 全局工具函数
├── api/                         # API 层 (请求封装 + 接口定义)
├── types/                       # 全局类型定义
├── pages/                       # 路由页面组件（薄层，组装 features）
├── styles/                      # 全局样式
├── App.tsx
├── main.tsx
└── vite-env.d.ts
```

### vs Bulletproof React

| 维度 | Bulletproof React | 本项目 | 差异说明 |
|------|------------------|--------|---------|
| features | ✅ 自包含模块 | ✅ 相同 | 完全对齐 |
| components | 共享组件 | ✅ 分类子目录 (ui/layout/feedback) | 更细分 |
| pages | app/routes | pages/ 目录 | 等价，命名不同 |
| stores | stores/ 目录 | features 内部 | 按 feature 就近放置 |
| api | features 内部 | 独立 api/ 目录 | 因为 RPC 封装需统一 |
| lib | lib/ 目录 | utils/ | 命名不同 |

## 五、ESLint 导入约束

### 禁止 features 互相导入

```js
// eslint.config.js
'import/no-restricted-paths': [
  'error',
  {
    zones: [
      // features 之间不能互相导入
      {
        target: './src/features/auth',
        from: './src/features',
        except: ['./auth'],
      },
      {
        target: './src/features/workbench',
        from: './src/features',
        except: ['./workbench'],
      },
      // 单向依赖：shared 不能导入 features/app
      {
        target: [
          './src/components',
          './src/hooks',
          './src/utils',
          './src/types',
        ],
        from: ['./src/features', './src/pages'],
      },
    ],
  },
],
```

### 强制使用 @/ 别名

```js
// eslint.config.js
'no-restricted-imports': [
  'error',
  {
    patterns: [
      {
        group: ['../*'],
        message: '禁止使用 ../ 相对路径，请使用 @/ 别名',
      },
    ],
  },
],
```

## 六、命名约定速查

| 类型 | 规则 | 示例 |
|------|------|------|
| 组件文件 | `PascalCase.tsx` | `Button.tsx` |
| 组件文件夹 | `PascalCase/` | `Button/` |
| 分类文件夹 | `kebab-case/` | `data-display/` |
| Hook 文件 | `use*.ts` | `useDebounce.ts` |
| 工具文件 | `camelCase.ts` | `formatDate.ts` |
| 类型文件 | `camelCase.ts` | `api.ts` |
| 测试文件 | `*.test.tsx` | `Button.test.tsx` |
| 常量文件 | `camelCase.ts` | `routes.ts` |
| 组件函数名 | PascalCase | `export function Button` |
| Props 类型 | 组件名+Props | `ButtonProps` |
| 事件回调 | on+事件 | `onClick`, `onSelectionChange` |
| 布尔状态 | is/has/should 前缀 | `isOpen`, `hasError` |

## 七、文件夹新增决策

```
要新增一个组件/功能，放哪里？

是否与业务逻辑相关？
  ├─ 否 → src/components/ (按分类放 ui/layout/feedback/data-display/data-input)
  └─ 是 → src/features/xxx/components/ (放在对应 feature 内)

是否被多个 feature 使用？
  ├─ 否 → 保持在单个 feature 内
  └─ 是 → 考虑提取到 src/components/ 或 src/hooks/

是 API 相关还是纯前端？
  ├─ API 请求 → src/api/ (因为 RPC 封装需统一)
  └─ 纯前端逻辑 → feature 内部或 src/hooks/
```
