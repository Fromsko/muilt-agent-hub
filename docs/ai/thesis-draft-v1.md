# 基于FastAPI与React的多用户AI智能体配置与交互平台设计与实现（初稿）

## 摘要

人工智能技术的高速发展，使得大模型应用逐步从单点调用进入平台化使用阶段。团队在接入模型服务后，不仅需要处理文本生成本身，还要管理提示词、模型密钥、智能体参数、用户会话、开放接口和运行日志。很多项目把这些能力拆散在脚本、环境变量和临时页面中，功能虽然能跑起来，后续维护却容易失控。配置难复用、权限边界不清、外部调用通道不统一、调用过程缺少审计，这些问题在多用户场景里会更加明显。针对这些问题，本文设计并实现了一个Fastapi+React的多智能体配置与交互平台。

平台采用前后端分离架构。前端基于React构建控制台界面，提供登录、仪表盘、Prompt管理、模型密钥管理、智能体配置、对话交互、API Token管理、OpenKey管理、MCP工具管理、日志查询、用户管理和系统设置等功能。后端基于FastAPI提供统一API服务，使用SQLModel定义数据模型，使用SQLAlchemy处理异步会话访问，使用SQLite存储平台数据，使用fastapi-users管理用户体系，使用JWT完成认证，使用LiteLLM统一接入不同大模型，使用Fernet加密保护敏感密钥，并通过平台API Token通道和OpenAI兼容网关通道为外部系统提供开放调用能力。

平台实现了智能体全生命周期管理，企业无需专业 AI 技术团队即可完成部署与使用，降低了 AI 应用的技术门槛和管理成本，适配中小公司的实际业务需求。

## 关键词

智能体管理平台；FastAPI；React；MCP；OpenAI兼容接口

## Abstract

The rapid development of artificial intelligence technology has led large-model applications to gradually shift from isolated API calls to platform-based usage. After integrating model services, teams need to manage not only text generation itself but also prompts, model keys, agent parameters, user sessions, external access channels, and runtime logs. Many projects scatter these capabilities across scripts, environment variables, and temporary pages. While the functionality may initially work, subsequent maintenance becomes difficult to control. Configuration reuse is challenging, permission boundaries are unclear, external access channels are inconsistent, and call processes lack auditing. These issues become even more pronounced in multi-user scenarios. To address these problems, this thesis designs and implements a multi-agent configuration and interaction platform based on FastAPI and React.

The platform adopts a frontend-backend separation architecture. The frontend console is built with React and provides login, dashboard, prompt management, model key management, agent configuration, chat interaction, API token management, OpenKey management, MCP tool management, log query, user management, and system settings. The backend provides unified API services based on FastAPI, using SQLModel and SQLAlchemy for data modeling and access, SQLite for platform data storage, fastapi-users for JWT authentication and user systems, LiteLLM for unified access to different large models, and Fernet for protecting sensitive keys. Through platform API tokens and an OpenAI-compatible gateway, the platform provides two types of external access capabilities for external systems.

The platform achieves full lifecycle management of intelligent agents, enabling enterprises to complete deployment and usage without a professional AI technical team, thereby reducing the technical threshold and management costs of AI applications and adapting to the actual business needs of small and medium-sized enterprises.

# 第一章 前言

## 1.1 项目背景和意义

大模型应用已经进入更细的工程化阶段[1]。团队不再满足于“能调用一次模型”这件事，而是希望把提示词、密钥、智能体配置、消息记录、权限控制和开放接口放进一套可管理的平台中。只有这样，系统才能被多人协作、重复调用和持续维护。模型供应商一多，配置项、鉴权方式和接口地址也会跟着变复杂。平台如果没有统一管理能力，开发人员很快就会陷入重复配置和重复排错。

从使用方式看，AI平台的服务对象也在变化。平台既要服务登录后的内部用户，也要服务外部系统接入。前者更关心页面体验、资源管理和对话效率，后者更关心稳定接口、权限边界和调用规范。单纯依赖后端脚本或临时接口，很难同时满足这两类需求。前端控制台如果缺失，用户对资源的创建、修改和追踪就不方便；后端平台如果缺少统一鉴权和日志机制，系统开放之后也容易出现失控问题。

本项目的意义主要落在两个方面。一个方面是工程组织意义。系统采用React控制台与FastAPI服务分离的方式，把登录、资源管理、对话交互、开放密钥、日志查询和用户管理放到同一套平台里协同实现。另一个方面是应用实践意义。系统提供JWT登录通道、平台API Token通道和OpenAI兼容通道，使内部用户和第三方系统都能使用平台能力。平台同时保留日志、统计和工具扩展机制，便于后续部署、排障和功能迭代。

## 1.2 研究开发现状分析

现有与大模型相关的系统大致可以分成三类。第一类重视模型统一接入，主要解决不同厂商接口不一致的问题。第二类重视智能体编排和工具调用，重点放在Prompt、工具链和任务执行流程[2][3]。第三类更偏向业务平台建设，会把账号、权限、日志、开放接口和管理页面一起纳入系统设计。三类方案都各有价值，但真正面向团队使用时，往往需要把它们的能力组合起来。

从当前项目的实现角度看，平台化落地还有几个明显难点。其一，多用户资源需要隔离，普通用户不能访问他人的Prompt、Key和Agent。其二，敏感信息不能明文保存，模型密钥和开放密钥都要有安全保护策略。其三，前端和后端之间不能只靠临时约定，登录态恢复、路由保护、请求拦截、错误提示和消息刷新都要形成稳定机制。其四，外部系统接入时需要标准化调用入口，否则平台能力就很难复用。其五，系统一旦接入外部工具，就必须保留可追踪的日志和调用记录。

本项目没有把这些问题拆成孤立模块分别处理，而是把它们放到同一套前后端系统里解决。前端控制台提供页面化的管理入口，后端平台提供统一的资源模型和调用能力。这样的设计更符合当前项目的真实形态，也更适合毕业设计对系统完整性、可运行性和工程闭环的要求。

## 1.3 主要研究内容

本项目围绕多用户AI智能体平台的前后端协同实现展开。前端部分构建了一套面向登录用户的控制台界面。用户进入系统后，可以在界面中完成登录、查看仪表盘、维护Prompt、录入模型密钥、创建智能体、进入对话页面、查看日志以及调整系统设置。管理员还可以在前端页面中管理用户和开放密钥。这样做的目的是把原本依赖接口操作的过程变成更容易使用和演示的页面流程。

后端部分负责提供统一的数据和业务能力。系统建立了用户、Prompt、Key、Agent、Message、ApiToken、OpenKey、McpServer、CallLog和AppLog等核心数据模型。用户登录后，可以把Prompt和模型密钥组合成Agent；对话请求到达后，后端会读取Agent配置、组装系统提示词和历史消息，再通过LiteLLM访问实际模型服务。消息和调用结果会被持久化保存，便于后续查看、统计和审计。

系统还实现了两类开放接入能力。一类是平台API Token，用于访问公开路由，适合自有系统或自动化流程直接调用。另一类是OpenKey，用于OpenAI兼容网关，适合第三方SDK按统一格式访问内部Agent。与此同时，系统还接入了MCP服务管理能力。用户可以登记外部工具服务，发现工具清单，再把工具绑定到指定Agent上，使模型在对话过程中具备调用外部能力。

# 第二章 相关技术与原理

## 2.1 Python与FastAPI后端框架

Python是一门解释型高级编程语言，语法简洁、生态丰富，在Web开发和人工智能领域有广泛应用。Python Web框架经历了Django、Flask、Tornado等阶段，各有侧重。FastAPI是近年出现的现代异步Web框架，基于Starlette和Pydantic构建，兼顾高性能与开发效率[4]。FastAPI的核心特性包括：自动生成OpenAPI文档、基于Pydantic的请求数据验证、原生异步支持，以及依赖注入机制。根据TechEmpower基准测试，FastAPI在吞吐量上接近Go和Node.js框架。

在本项目中，FastAPI承担后端服务的核心职责。系统通过FastAPI创建应用实例，再通过APIRouter组织业务路由。依赖注入机制在认证、数据库会话管理和日志上下文传递中发挥了关键作用，例如路由函数通过声明依赖即可自动获取当前登录用户和数据库会话对象，避免了重复的上下文获取代码。FastAPI的异步支持使系统在处理模型调用、MCP工具调用等IO密集型操作时保持较高并发能力。此外，系统启动后可直接通过Swagger UI测试各个接口，降低了开发和演示阶段的调试成本。

## 2.2 React前端框架

React是由Meta于2013年开源的JavaScript前端框架，核心思想是将界面拆分为可复用的组件。React采用虚拟DOM技术和声明式编程范式，通过比较新旧虚拟DOM差异来最小化实际DOM操作，提高渲染性能。React引入的Hooks机制使函数组件也能管理状态和生命周期，常用的useState、useEffect、useContext等简化了组件逻辑的组织方式。

本项目使用React构建控制台界面，将登录页、仪表盘、Prompt管理页、模型密钥管理页、Agent管理页、对话页、日志页、用户管理页和系统设置页拆分为独立组件，再通过统一布局组合成完整工作台。项目使用Ant Design作为基础组件库，表格、表单、弹窗、抽屉等控件均来自该体系，页面结构统一、交互反馈稳定。前端构建方面，项目采用Rsbuild工具，基于Rust实现，配置简单且构建速度快，支持热模块替换、TypeScript和CSS Modules等特性，能够满足毕业设计项目的开发和部署需求。

## 2.3 前端路由与状态管理

前端路由是单页应用的核心机制。本项目采用TanStack Router管理路由结构，该方案基于文件系统路由，支持类型安全的路由参数。系统在受保护路由入口中先检查登录状态，再根据路径和权限判断是否允许访问目标页面，未登录用户重定向到登录页，权限不足用户引导到403页面。这种处理方式把登录保护和权限校验前置到路由层，避免了每个页面重复编写权限检查逻辑。

状态管理方面，项目将远程状态和本地状态分开处理。TanStack Query负责管理服务端返回的数据状态，把Prompt列表、密钥列表、智能体列表等数据统一交给Query处理，通过缓存机制减少重复请求，资源变更后主动刷新相关查询。Zustand负责管理本地会话状态，包括登录令牌、当前用户信息、菜单权限和界面偏好。这种分工避免了单一状态层过于臃肿，也便于后续扩展。

## 2.4 数据模型与持久化

SQLModel是本项目主要的数据模型定义工具，由FastAPI作者开发[5]。其核心思想是"一个模型，两份用途"：同一个模型类既可作为Pydantic模型用于请求验证，也可作为SQLAlchemy表模型用于数据库操作，避免了重复定义字段的问题。项目中的Prompt、Key、Agent、Message、ApiToken、OpenKey、McpServer、CallLog和AppLog等资源都采用SQLModel定义，字段类型、约束和外键关系均通过类型注解声明。

数据库访问层基于SQLAlchemy异步会话实现，系统通过统一的AsyncSession工厂管理连接，在应用启动时自动完成建表。当前项目默认使用SQLite作为数据存储，它是嵌入式关系型数据库，不需要独立服务器进程，数据存储在单个文件中，部署成本低，适合毕业设计阶段的快速部署和本地演示。

## 2.5 JWT认证与用户体系

JWT（JSON Web Token）是一种无状态认证方案[6]，将用户信息编码为JSON格式并经过签名生成令牌。客户端在请求中携带JWT令牌，服务器验证签名即可解析用户信息，无需查询会话存储。JWT由头部、载荷和签名三部分组成，优点是无状态、可跨域，缺点是签发后无法服务端撤销，通常通过设置较短过期时间并配合刷新令牌机制来解决。

本项目的用户体系基于fastapi-users实现，该库封装了用户模型适配、注册登录和JWT认证等通用能力。项目在此基础上补充了刷新令牌接口，使客户端在令牌即将过期时可以静默续期。系统区分普通用户和超级管理员两类角色，普通用户管理自己的资源，管理员可管理平台用户并签发OpenKey。权限控制由前端受保护路由和后端资源归属校验共同完成，形成前后端双重校验机制。

## 2.6 LiteLLM统一模型接入

大语言模型服务提供商众多，不同提供商的接口、鉴权方式和参数格式各不相同。LiteLLM是一个开源的统一模型调用库，屏蔽不同提供商之间的差异，提供统一调用接口[7]。它支持OpenAI、Anthropic、DeepSeek、智谱AI等数十个提供商，也支持Ollama等本地模型服务，并提供流式和非流式响应、重试机制和成本跟踪等特性。

在本项目中，系统在Agent配置中记录模型名、温度和最大输出长度，在Key资源中保存访问密钥和可选的接口地址。聊天服务在发起调用前将这些配置组装成统一参数交给LiteLLM执行，系统同时支持流式和非流式两种响应方式。用户还可以通过配置Ollama提供商接入本地开源模型，实现云端商业模型与本地模型的灵活切换。

## 2.7 智能体技术与MCP协议

智能体（Agent）是指能够感知环境、做出决策并执行动作的自主实体[2][8]。在大语言模型时代，智能体基于LLM理解自然语言指令、规划任务步骤、调用外部工具并处理多轮对话。智能体的核心组件包括感知模块、决策模块、记忆模块和工具模块，它们协同形成感知-决策-行动循环。智能体区别于单次提示词调用的关键在于动态提示词构建和工具调用能力：现代LLM支持在生成文本的同时输出结构化的工具调用请求，系统执行工具函数后将结果返回模型继续生成。

MCP（Model Context Protocol）是由Anthropic提出的协议[3]，用于在LLM应用中统一管理工具和上下文。MCP定义了工具发现、工具调用和结果返回等标准流程，支持HTTP、stdio等传输方式，其核心优势是解耦工具提供者和使用者。在本项目中，用户可在前端页面登记MCP服务，通过"发现工具"功能拉取工具清单，再把工具绑定到指定Agent。聊天服务运行时读取绑定关系，将工具描述转换为模型可识别格式，模型发起工具调用时系统找到对应MCP服务执行并将结果写回上下文。本项目中的Agent把Prompt、模型密钥、推理参数和MCP工具绑定关系聚合到同一实体中，实现了从"单条消息调用模型"到"可重复配置的智能体调用"的提升。

## 2.8 安全存储机制

平台涉及多种敏感信息，需要采用不同的安全策略。对于需要还原的信息（如模型API密钥），系统采用Fernet对称加密。Fernet基于AES-128-CBC算法，系统在密钥入库前调用加密函数生成密文，在调用模型时解密还原明文，加密密钥通过环境变量配置，不硬编码在代码中。

对于不需要还原的凭证（如平台API Token和OpenKey），系统采用哈希化方式存储。创建凭证时生成随机字符串作为明文值并计算其哈希值存入数据库，后续验证通过比较哈希值完成。系统只在创建时返回一次明文值，后续查询仅显示脱敏结果。OpenKey还支持调用额度和分钟级限频控制，减少外部滥用风险。

## 2.9 本章小结

本章说明了本项目的技术基础。后端由Python、FastAPI、SQLModel和SQLite组成，前端由React、TanStack Router、TanStack Query、Zustand和Rsbuild组成，模型接入通过LiteLLM统一处理，工具扩展通过MCP协议实现，安全存储由Fernet加密和哈希化机制保障。以上技术共同支撑后续系统分析、系统设计和系统实现章节。

# 第三章 系统分析

## 3.1 可行性分析

### 3.1.1 技术可行性

从当前代码实现看，本项目已经具备较完整的技术落地基础。前端已经形成可运行的React控制台，包含登录、仪表盘、Prompt管理、模型密钥管理、智能体管理、对话页面、API Token页面、OpenKey页面、MCP工具页面、日志页面、用户管理页面和系统设置页面。前端路由、认证状态恢复、请求封装和资源查询机制都已经打通，页面并不是静态原型，而是与后端接口直接联动的工作台。后端则基于FastAPI提供统一API服务，已经具备用户认证、资源隔离、模型调用、消息持久化、统计查询、日志审计和开放网关等能力。现有实现已经覆盖系统的主要业务链路，因此本项目不存在“只有设计、没有实现”的问题。

系统所依赖的关键技术也比较成熟。React、TanStack Router、TanStack Query、Zustand和Ant Design已经被广泛用于中后台和控制台类系统；FastAPI、SQLModel、SQLAlchemy和fastapi-users在Python Web开发中也有较稳定的工程实践。模型调用部分交给LiteLLM统一处理，可以减少多厂商接口差异带来的开发成本；工具扩展部分通过MCP协议接入，避免把工具调用逻辑硬编码在聊天主流程中。从技术选型和现有代码成熟度看，本项目具备较高的技术可行性[6][4]。

### 3.1.2 经济可行性

本项目的建设成本整体较低。前后端使用的核心技术栈都以开源框架和组件库为主，开发环境不需要额外采购昂贵的商业软件。前端依赖Node生态即可完成开发和打包，后端依赖Python环境即可启动运行，数据库默认使用SQLite，这使得项目在课程设计、毕业设计和答辩演示阶段可以快速搭建起来。对于学校环境或个人电脑而言，这样的部署门槛是可以接受的。

模型调用虽然会带来一定外部API成本，但系统在设计上保留了较强的灵活性。用户可以通过模型密钥页面配置不同厂商的密钥和兼容接口地址，也可以接入本地兼容服务。这样一来，系统既可以在演示阶段使用轻量模型控制成本，也可以在正式应用场景中接入性能更强的模型服务。平台还把内部使用和外部调用统一纳入同一套后端能力中，减少了重复建设多套服务的开销，因此项目在经济上是可承受的。

### 3.1.3 运行可行性

系统的运行方式比较清晰。登录用户通过浏览器访问React控制台，完成资源管理和对话交互；第三方系统可以通过平台API Token访问公开接口，也可以通过OpenKey调用OpenAI兼容网关。三类使用方式共享同一套核心后端能力，但认证方式和使用入口相互区分，这有利于控制边界并减少混乱。对普通用户而言，系统的操作路径比较直接，页面与接口一一对应；对开发人员而言，系统的日志和统计能力也便于观察运行状态。

现有实现还具备一定的运行保障基础。后端提供调用日志、应用日志和统计接口，前端仪表盘可以展示资源总览和最近7天调用趋势，日志页支持按级别、logger和关键字过滤结果。项目中已经配备冒烟测试脚本，前端工程也配置了测试脚本和端到端测试入口。这说明系统并不是只能在理想场景下演示，而是已经开始考虑运行验证和故障排查，因此在运行层面具备可行性。

### 3.1.4 维护可行性

本项目的模块边界比较清楚，这为后续维护提供了基础。前端按路由和资源组织页面，登录、Prompt、模型密钥、智能体、对话、日志、用户管理等功能都放在独立页面中。后端则按模型、路由、服务和认证模块拆分代码，资源模型和业务入口之间的关系清晰。开发人员定位问题时，可以较快判断应该查看前端页面逻辑、HTTP客户端、后端路由、服务函数还是数据库模型，不需要在单一大文件中反复排查。

系统的扩展路径也比较明确。若后续增加新的资源类型，可以参考现有Prompt、Key、Agent这类模块继续扩展；若后续增加新的模型服务，可以沿着LiteLLM统一接入方式补充配置；若后续增加新的工具能力，可以通过MCP服务注册与绑定机制接入。系统把共性的认证、日志、统计和开放接口抽出来单独处理，避免每新增一个功能就重新实现同样的基础逻辑，因此维护成本相对可控。

## 3.2 需求分析

本项目的需求来源并不单一。系统既要服务登录后的内部用户，也要服务拥有授权凭证的外部调用方，还要服务负责平台治理的管理员。三类主体关注的问题并不相同，但都需要围绕同一套资源模型协同工作。系统因此不能只停留在“聊天接口可用”的层面，而要把认证、页面交互、资源管理、开放接口、日志审计和权限控制一起纳入需求范围。

### 3.2.1 功能需求

对于普通用户而言，系统需要提供完整的控制台使用能力。用户进入平台后，应当能够完成登录认证、恢复已有会话、查看仪表盘中的资源数量与最近消息、创建和维护Prompt模板、录入模型密钥、创建智能体并配置模型参数、进入指定智能体的对话页面、查看历史消息，以及在系统设置页面调整界面偏好。由于这些资源都具有明显的私有属性，系统还需要确保用户只能看到和操作属于自己的Prompt、Key、Agent和消息记录。资源操作不仅要能创建，也要支持列表查询、编辑更新和删除处理，才能满足日常使用需求。

对于管理员而言，系统还需要提供平台治理能力。管理员除了拥有普通用户的基本使用能力，还需要能够查看和维护平台用户、签发OpenKey、控制开放调用范围、观察日志和统计信息，并在必要时禁用或删除问题凭证。用户管理功能要求系统区分普通用户和超级管理员两类身份，防止普通用户进入管理页面。开放密钥管理功能则要求管理员能够为外部系统设置可访问的Agent范围、调用频率和总调用额度，从而把开放能力放在受控边界之内。

对于外部调用方而言，系统需要提供标准化接入通道。一条通道是平台API Token，主要面向平台公开路由，适合自有服务、Webhook、自动化流程或轻量集成场景；另一条通道是OpenAI兼容网关，适合使用现有OpenAI SDK的外部系统。两条通道都需要让调用方在不进入前端控制台的情况下完成消息发送和结果获取，同时又要保证调用范围、调用频率和调用记录都可追踪。除此之外，系统还需要支持MCP工具注册、工具发现和Agent工具绑定，使智能体在对话阶段具备调用外部能力，这也是平台区别于纯聊天接口的重要功能需求。综合以上三类角色的需求，系统用例图如图3-1所示。

### 3.2.2 非功能需求

系统在非功能层面需要满足可用性、可维护性和扩展性要求。可用性要求前端页面结构清晰、资源操作路径明确、错误提示可理解、加载状态有反馈，避免用户在使用控制台时频繁依赖开发人员解释。后端接口则需要具备较稳定的响应结构，便于前端统一封装请求和错误处理。对话场景中，系统还要支持流式输出，这样用户才能在页面中实时观察生成过程，而不是长时间等待一次性结果。

可维护性和扩展性要求系统保留清晰的层次边界。前端要能在不大幅修改已有页面的情况下继续增加资源页或管理页，后端要能在不破坏主链路的情况下接入新的模型服务、凭证类型或工具服务。系统还应具备一定的兼容性，既能服务浏览器中的控制台用户，也能服务通过HTTP或SDK接入的第三方系统。这些要求虽然不直接体现在某一个按钮或某一个接口上，却会直接影响系统后续是否能够持续演进。

### 3.2.3 安全性需求分析

安全性需求是本项目的核心约束之一。用户登录后访问平台资源，需要通过JWT进行身份验证；前端受保护路由只能在已登录且具备相应权限的情况下进入页面；后端在处理资源查询和更新时，还要根据当前用户身份继续校验资源归属，防止前端绕过界面限制直接访问他人数据。也就是说，系统不能把安全控制只放在页面层，也不能只依赖用户自行遵守调用规则，而是要在前后端共同建立访问边界。

敏感信息保护也属于安全性需求的重要部分。模型密钥不能明文存储在数据库中，因此系统需要在入库前进行Fernet加密；平台API Token和OpenKey属于平台级凭证，系统不能在数据库中保存明文，而应采用哈希化方式存储，并且只在创建成功时返回一次明文值。外部开放接口还需要支持启停控制、额度限制和频率限制，避免平台能力在暴露后被无约束地调用。除此之外，系统还需要保留调用日志和应用日志，使异常请求、失败调用和可疑行为能够被追踪和复盘。

### 3.2.4 数据流程分析

从数据流角度看，系统至少包含四条核心链路。登录链路中，用户在前端登录页输入邮箱和密码，前端向认证接口发送请求并获取访问令牌，再通过会话恢复逻辑拉取当前用户和权限信息，随后把用户状态写入本地存储并进入受保护页面。资源配置链路中，用户在Prompt页、模型密钥页和智能体页完成信息录入，前端通过统一HTTP客户端访问后端接口，后端完成数据校验、权限判断和数据库写入，再把结果返回前端页面展示。

对话链路中，前端会先拉取指定Agent信息和历史消息，再由用户在对话页面提交新消息。后端收到请求后，需要加载Agent配置、Prompt内容、消息历史、模型密钥和工具绑定信息，组装完整上下文后再调用LiteLLM。模型返回结果后，系统会把用户消息、助手回复和调用日志持久化保存，前端则把流式或非流式结果渲染到对话界面。开放接口链路中，外部系统不经过前端页面，而是直接携带API Token或OpenKey访问公开接口；后端在校验凭证、额度和可访问范围后，仍然会进入同一套核心对话服务，这样可以保证内部调用和外部调用在核心能力上保持一致。

## 3.3 本章小结

本章从技术、经济、运行和维护四个角度分析了系统的可行性，并结合普通用户、管理员和外部调用方三类主体梳理了功能需求、非功能需求、安全需求和数据流程需求。可以看出，本项目的关键难点不在单次模型调用，而在于如何把控制台页面、后端服务、开放接口、工具扩展和运行审计组织成一套稳定系统。后续系统设计章节将在这些需求基础上展开。

需要注意一点：第三章在需求分析阶段将系统功能抽象为六个业务模块，分别为认证入口、资源配置、对话执行、开放接入、工具扩展与平台治理。第四章至第六章在设计实现测试部分，为使叙述更贴近用户实际使用流程，章节标题按登录、配置、对话、开放调用、工具接入、平台管理这一业务逻辑顺序编排。各章开头会给出模块之间的对应关系，确保全文逻辑连贯。

### 4.2.1 登录认证与会话恢复设计

用户认证模块由前端登录页、认证状态仓库、会话恢复逻辑和后端JWT认证接口共同组成。用户在登录页提交邮箱和密码后，前端调用认证接口获取访问令牌，并尝试获取刷新令牌。令牌写入本地持久化状态后，前端会立即调用当前用户信息接口和权限接口恢复会话，再根据权限生成可访问菜单。此后，受保护路由在用户进入页面前再次校验登录状态和页面权限。这样的设计让登录成功后的页面切换、刷新恢复和权限控制形成完整闭环。

后端在这一模块中的职责，是保证认证结果可信且可继续续期。系统基于fastapi-users维护用户体系，并补充刷新令牌能力，使客户端在一定时间内不必反复登录。认证通过后，普通用户可以管理自己的资源，管理员则拥有额外的平台治理权限。前后端共同参与认证控制的原因在于，仅靠前端限制不足以保证安全，仅靠后端校验也不足以保证页面体验，二者结合更适合当前项目。登录认证流程如图4-4所示。

### 4.2.3 Prompt、模型密钥与智能体聚合配置设计

资源配置模块是平台的核心基础模块，通过整合不同的分散资源，形成可直接复用的智能体。

#### 4.2.3.1 Prompt管理设计

Prompt用于保存系统提示词模板。在设计上，提示词作为独立资源实体进行存储，与具体模型解耦。用户可以创建、更新和分类管理多种应用场景的提示词模板。系统在后台为每个Prompt设置用户归属权，保证资源的私密性与安全隔离。

#### 4.2.3.2 模型密钥管理设计

模型密钥管理用于安全保存各类大模型服务商（如OpenAI、Anthropic、Ollama等）的访问凭证。设计时考虑到凭证的敏感性，所有密钥信息在写入数据库前进行对称加密，并且前端列表仅展示脱敏后的尾缀字符。模块还设计了最小化网络探测机制，允许在录入时即刻验证密钥与接口的连通性。

#### 4.2.3.3 Agent聚合配置设计

智能体设计把Prompt、模型密钥、模型名称、温度（默认设定为0.7以保证生成多样性与合理性的平衡）和最大输出长度（默认设定为2048 Tokens）等参数组织到同一实体中。通过这一聚合设计，平台实现了从“单条消息调用模型”到“可重复配置的智能体调用”的提升。此外，Agent还支持绑定MCP工具，使单纯文本生成能力扩展为可调用外部函数的复合执行单元。资源配置模块的处理流程如图4-5所示。

### 4.2.4 对话页面、流式响应与消息历史设计

## 4.4 接口设计

系统接口按功能可以划分为认证接口组、资源配置接口组、对话接口组、开放接口组和平台治理接口组。认证接口组主要包括登录、刷新令牌和当前用户信息接口；资源配置接口组主要包括Prompt、模型密钥、智能体、MCP服务、API Token和OpenKey等资源的创建、查询、更新与删除接口；对话接口组主要包括内部聊天接口和消息历史接口；开放接口组主要包括平台公开聊天接口和OpenAI兼容接口；平台治理接口组则包括统计接口、日志接口和管理员用户管理接口。这样的接口分组与前端页面分工较为一致，便于前后端协同开发。

从调用方式看，系统接口同时支持标准JSON请求和流式响应。普通资源接口大多采用基于/api/v1前缀的REST风格路径，前端通过统一HTTP客户端携带JWT访问这些接口；公开调用接口根据凭证类型分别校验平台API Token或OpenKey；对话接口在流式模式下采用SSE返回增量内容。以当前项目为例，/api/v1/auth/jwt/login负责登录，/api/v1/users/me返回当前用户信息，/api/v1/agents/{agent_id}/chat负责内部对话，/api/v1/public/agents/{agent_id}/chat负责平台Token对话，/v1/chat/completions负责OpenAI兼容调用，/api/v1/logs和/api/v1/stats分别负责日志与统计查询。接口设计既保留了平台自身的资源语义，也兼顾了外部系统接入的兼容性。

## 4.5 本章小结

本章围绕系统总体架构、主要功能模块、数据库结构和接口组织方式展开设计说明。可以看到，本项目并不是把前端页面和后端接口简单拼接在一起，而是围绕统一资源模型和统一调用链路进行组织。React控制台承担用户操作入口和状态管理，FastAPI后端承担核心业务处理和开放能力输出，SQLite、大模型服务和MCP工具服务则为平台提供数据和外部能力支撑。这样的设计为后续系统实现和测试章节提供了清晰基础。

# 第五章 系统实现

本项目的实现重点，并不是去做一个面向海量租户、复杂计费和大规模商业运营的平台，而是面向中小型企业的内部AI能力整合场景。中小企业在引入大模型时，最常见的问题不是“模型能不能调用”，而是Prompt分散在文档里、模型密钥分散在环境变量里、MCP工具各自独立运行、业务系统又要重复拼装Agent。这样做短期能跑，长期很难维护。本项目第5章的实现重点，正是把这些分散资源统一收拢到平台里，由平台直接接管prompt、key、agent和mcp，再把已经聚合好的Agent继续作为内部使用入口和外部调用入口输出出去。

用户进入某个Agent的对话页后，前端会先获取当前Agent信息和历史消息，再把这些内容交给运行时组件处理。前端运行时组件会先把历史消息按消息ID升序整理，再转换成对话线程初始数据，以保证页面中展示的消息顺序与真实会话顺序一致。这个细节虽然不大，但直接影响用户对平台可用性的感受。如果历史消息顺序混乱，平台就很难被持续使用。智能体对话页面如图5-14所示。

<!-- IMAGE:5-14:智能体对话页面 -->

发送消息时，前端主要通过流式接口调用后端对话服务，并使用SSE方式接收增量输出。前端内部逐帧解析服务端推送的增量数据，把服务端返回的delta逐段渲染到页面上。对用户而言，这意味着平台不是在发送消息后长时间无反馈，而是能够实时观察模型的生成过程。对于中小企业场景来说，这种流式交互比复杂的会话编排界面更重要，因为它直接决定平台是否“好用”。

后端对话能力则统一收口在核心对话服务中。无论请求来自前端对话页、平台公开接口还是OpenAI兼容网关，都会进入同一套核心处理流程：加载Agent配置、读取Prompt、读取历史消息、读取密钥、组装上下文、调用LiteLLM、保存用户消息和助手回复，并记录调用日志。这样一来，前端页面与外部系统虽然入口不同，但最终消费的都是同一个聚合后的Agent能力，而不是互不相干的多个实现分支。这也正是平台化落地优于脚本化调用的地方。

核心对话服务的主入口函数do_chat的实现如图5-15所示。该函数首先加载Agent上下文（包括关联的Prompt和模型密钥），再获取历史消息构建完整对话序列。如果Agent绑定了MCP工具服务，则并发发现所有可用工具并注入LiteLLM调用参数。对于流式请求，系统通过SSE生成器逐帧推送模型输出，并在完成后异步持久化消息和调用日志。

<!-- IMAGE:5-15:核心对话服务执行逻辑代码 -->

前端流式对话的实现采用适配器模式，如图5-16所示。前端将后端SSE流桥接到assistant-ui的ChatModelAdapter接口，从对话历史中提取最后一条用户消息发送到后端，再逐帧解析服务端推送的增量数据并拼接渲染到页面上。这种设计使前端对话组件与后端通信协议解耦，后续替换通信方式不需要修改UI层代码。

## 5.5 聚合能力的外部调用实现

### 5.5.1 平台API Token调用的实现

平台API Token管理页面如图5-17所示，用户可以为某个外部场景创建Token，例如Webhook、自动化流程或企业内部其他服务。Token创建成功后，系统只在当次弹窗中展示一次明文值，后续列表中只保留前缀、尾缀和启停状态。这与模型密钥一样，体现了平台对敏感凭证的谨慎处理。前端页面还直接给出了调用示例，说明外部系统可以使用Authorization: Bearer <token>的方式访问/api/v1/public/agents/<AGENT_ID>/chat接口。

<!-- IMAGE:5-17:平台API Token管理页面 -->

这一能力非常适合中小企业内部集成。中小企业往往没有复杂的统一API网关和认证平台，但又确实需要让某些自动化流程、表单系统、知识库机器人或内部应用接入AI能力。平台API Token的作用，就是把“谁能访问哪个Agent”这件事放回平台来管理，而不是让每个业务系统自己保存模型密钥和Prompt配置。业务系统只需要调用聚合后的Agent，不需要直接接触底层模型凭证。

### 5.5.2 OpenKey与OpenAI兼容网关的实现

除了平台Token，本项目还实现了OpenKey与OpenAI兼容网关。OpenKey管理页面如图5-18所示，管理员可以在页面中创建OpenKey，并设置允许访问的Agent范围、每分钟调用限制、总额度限制和过期时间。页面同样只在创建时展示一次完整明文Key，之后只展示脱敏信息。这样的实现使平台开放能力处于可控状态，而不是把一个永久有效的共享凭证散发给所有外部系统。

<!-- IMAGE:5-18:OpenKey管理页面 -->

后端实现了/v1/chat/completions和/v1/models接口。调用方使用OpenKey认证后，可以通过model="agent-<id>"的方式把请求映射到平台内部某个Agent。这里最关键的一点是，外部系统调用的不是裸模型，而是已经在平台内聚合好的Agent。也就是说，Prompt、模型密钥、模型参数，甚至后续绑定的工具能力，都由平台统一管理，外部系统只需要按标准OpenAI协议调用即可。这正是平台聚合价值向外复用的具体体现。OpenAI兼容网关的核心接口代码如图5-19所示，接口从请求的model字段解析出agent-id格式的Agent标识，校验OpenKey的Agent访问权限后将请求转化为内部ChatRequest调用统一的do_chat服务，返回时再将内部响应格式转换为OpenAI标准的chat.completion结构。

<!-- IMAGE:5-19:OpenAI兼容网关核心接口代码 -->

对于中小企业来说，这种实现比“让每个系统自己接OpenAI”更实用。因为企业真正需要的不是再多一套模型调用代码，而是把已经在平台中沉淀好的能力稳定地复用出去。平台把Agent作为对外输出单元后，企业内部不同系统就能围绕统一的能力资产协作，而不是重复建设。

## 5.6 MCP工具接管与绑定的实现

MCP工具管理页面如图5-20所示，页面支持两种接入方式：HTTP方式和stdio方式。用户创建MCP服务时，需要填写服务名称、连接信息和可选的Bearer Token。服务创建完成后，用户可以点击“发现工具”，平台会临时连接该MCP服务并拉取工具清单，再把工具名称、描述和输入结构显示在抽屉中。这个过程不把工具清单落库，而是先让用户确认当前服务是否真的可用。

<!-- IMAGE:5-20:MCP工具管理页面 -->

后端提供了MCP服务CRUD、工具发现以及Agent工具绑定接口。Agent与MCP服务之间采用覆盖式绑定策略，更新绑定时会先清空旧关系，再写入新的中间表记录。这样做比较直接，也更适合当前项目规模。对于中小企业内部平台而言，这种做法已经足够稳妥，因为平台重点在于“把工具能力纳入统一控制”，而不是追求极度复杂的工具编排能力。MCP工具管理器的核心代码如图5-21所示，连接部分如图5-22所示。管理器封装了HTTP和stdio两种MCP连接方式，HTTP连接使用连接池复用会话，通过LRU缓存和TTL机制避免频繁握手，连接失败时自动清除缓存条目。工具发现接口通过MCP协议拉取服务端工具清单，再转换为LiteLLM期望的JSON Schema格式注入模型调用参数。工具执行接口负责调用指定工具并将结果文本返回给模型继续推理。

<!-- IMAGE:5-21:MCP工具管理器核心代码 -->

<!-- IMAGE:5-22:MCP工具管理器连接代码 -->

MCP能力被纳入平台后的实际价值很明显。企业内部不再需要让每个业务系统单独维护一个MCP客户端，也不需要把工具接入逻辑散落在不同脚本里。平台统一接管MCP服务，用户只需要决定“哪个Agent绑定哪些工具”。在对话执行阶段，后端再根据Agent绑定关系完成工具发现和工具调用。这种模式把工具接入从分散实现变成平台能力，是本项目区别于普通聊天页面的重要实现点。

## 5.7 日志、用户管理与系统设置的实现

日志查看页面如图5-23所示，页面通过查询后端日志接口，支持按日志级别、logger名称和关键字进行过滤，也支持自动刷新和日志详情查看。后端同时区分调用日志与应用日志：调用日志关注模型调用过程和结果，应用日志关注请求链路与异常信息。这样的实现很符合中小企业平台运维的实际情况。中小企业通常没有独立的可观测平台团队，很多问题都要靠开发人员和管理员自己定位。把日志查询能力直接做进平台，能明显降低排障门槛。

## 5.8 本章小结

本章围绕登录认证、仪表盘、Prompt管理、模型密钥管理、Agent聚合配置、对话页面、外部调用、MCP接管以及平台治理功能，说明了当前项目的具体实现方式。可以看到，本项目最重要的实现价值并不在于再做一个聊天页面，而在于把中小企业内部最容易分散的几类核心资源——prompt、key、agent和mcp——统一接入平台并聚合起来。平台内部用户可以直接围绕聚合后的Agent使用系统，外部系统也可以通过平台API Token或OpenAI兼容网关继续调用这些已经聚合好的能力。这种“先平台内聚合，再统一向内外复用”的实现路径，正是本项目的核心特色。

## 6.1 测试目标与范围

本项目当前阶段的测试重点，不是面向大型商业平台的高并发压测，也不是去验证复杂的多租户计费逻辑，而是围绕中小型企业内部平台的真实需求，验证系统是否已经形成完整可用闭环。具体来说，测试目标主要包括四个方面：第一，验证前后端核心功能是否能够打通，包括登录、资源管理、Agent配置、对话交互和页面展示；第二，验证平台是否真正实现了资源隔离，确保不同用户之间无法互相读取或修改彼此的数据；第三，验证平台对外开放调用是否可用，确认聚合后的Agent能够被内部页面和外部系统共同复用；第四，验证项目是否具备基本部署条件，能够在常规开发环境中快速启动并提供演示与联调能力。

## 6.4 开放调用与集成验证

API Token和OpenKey的基本管理功能（创建、编辑、删除、权限校验）已在6.3节中验证，本节重点验证外部系统通过这两类凭证实际调用平台能力的集成场景。

第一类外部调用是平台API Token方式。用户创建Token后，外部系统可通过 /api/v1/public/agents/{agent_id}/chat 访问指定Agent。验证时，使用有效Token发送请求，系统能正确加载Agent配置（含Prompt、模型密钥等），返回模型回复。使用无效Token或停用的Token则返回401/403错误。

第二类外部调用是OpenAI兼容方式。后端通过 /v1/chat/completions 和 /v1/models 提供兼容接口。外部系统将 model 字段设置为 agent-<id> 即可调用内部Agent。验证时，使用有效OpenKey发送请求，系统返回符合OpenAI Chat Completion格式的响应；超出额度或频率限制时返回相应错误码。

集成验证结果表明，平台聚合后的Agent能力可以被外部系统稳定复用，外部调用与内部页面调用共享同一套核心对话服务，保证了行为一致性。

## 6.5 部署可行性与运行验证

从部署角度看，本项目更强调轻量、直接和可复制。后端以FastAPI为核心，默认使用SQLite作为数据存储，项目初始化时会自动建表并在没有管理员用户时创建默认管理员账号。这种设计方便在试验环境、教学环境和企业内部小规模试点环境中快速落地。前端以React控制台方式运行，既可在开发模式下联调，也可打包为静态资源后部署。整体部署过程不依赖复杂的容器编排、服务网格或分布式数据库。

# 前端构建与预览

对于中小企业的内部试点，该部署方案已能满足初期使用需求。若后续企业用户数量和调用量持续增长，可将数据库迁移到PostgreSQL等生产级方案，并增加反向代理（如Nginx）和进程管理工具（如Gunicorn）。

# 第七章 总结与展望

## 7.1 工作总结

本文围绕一个面向中小型企业内部使用的多用户AI智能体配置与交互平台，完成了需求分析、系统设计、系统实现以及测试与部署说明。项目采用FastAPI与React前后端分离架构，围绕企业内部最容易分散的四类核心资源——prompt、key、agent和mcp——构建平台能力，把原本散落在脚本、环境变量和零散服务中的配置与调用流程收拢到统一控制台和统一后端服务中。

与很多只强调模型调用的项目不同，本项目更关注“能力聚合后如何持续复用”。系统先把Prompt模板、模型密钥、智能体配置和MCP工具接入统一纳管，再通过内部控制台供用户直接使用，同时通过平台API Token和OpenAI兼容网关向外部系统继续开放。这样一来，企业内部不同人员和不同系统不需要各自维护一套模型接入流程，而是围绕平台已经聚合好的Agent能力协同工作。这一点正是本项目最核心的工程价值。

从实现结果看，系统已经具备登录认证、会话恢复、资源管理、对话交互、消息历史、外部调用、工具接入、日志审计、统计展示和管理员治理等能力。无论从前端页面完整性、后端接口结构，还是从资源隔离、安全保护和平台开放能力来看，项目都已经超出“简单聊天演示”的层面，具备了可落地、可扩展、可继续维护的内部平台雏形。

## 7.2 不足与改进方向

虽然当前项目已经完成了较完整的主链路，但仍然存在进一步优化空间。第一，数据库目前默认使用SQLite，更适合开发、演示和小规模试点环境。若后续企业用户数量和调用量持续增长，可以将数据库迁移到更适合生产环境的方案，以提升并发能力和运维弹性。第二，当前权限模型以普通用户和管理员为主，已经能满足中小企业内部平台的初步治理需求，但若后续涉及部门级分工、审批流或更细粒度的资源共享，仍需要继续扩展角色与权限体系。

第三，当前自动化测试基础已经具备，但测试覆盖仍可以继续加强。后端已经有冒烟测试，前端也已经配置了单元测试和端到端测试入口，后续可以围绕登录续期、流式对话、MCP工具绑定、平台Token调用和OpenAI兼容调用等关键链路补充更细粒度测试。第四，平台目前已经支持日志与统计，但在可观测性方面仍可继续增强，例如增加更细的调用看板、失败分类分析和Agent级别的运行画像，帮助企业更直观地判断平台使用效果。

第五，MCP能力虽然已经被统一接入平台，但当前仍以服务登记、工具发现和Agent绑定为主。后续可以在保持平台统一接管思路不变的前提下，继续增强工具权限管理、工具调用记录和工具分类组织能力。这样既能保持当前“先聚合再复用”的优势，也能让平台在企业实际使用中承载更多复杂场景。

总体来看，本项目已经验证了一条比较务实的落地路径：对于中小型企业而言，建设AI平台不一定要从庞大复杂的商业化架构开始，而可以先从统一接管prompt、key、agent和mcp这些最容易分散的核心资源开始，再把聚合后的能力稳定地提供给内部页面和外部系统使用。只要平台能够把资源收口、能力复用和运行治理这三件事做好，就已经具备较强的实际价值。后续工作将继续围绕这一方向展开[10]，使平台在保持轻量可控的同时，逐步提升稳定性、治理能力和集成深度。
